# Этап 1: Создание проекта

## Цель этого этапа

Создать пустой проект s&box с правильной структурой файлов. После этого этапа ты сможешь открыть проект в редакторе s&box.

## Шаг 1.1: Создание проекта в редакторе s&box

1. Открой **s&box** через Steam
2. В главном меню нажми **"Create a Project"** (Создать проект)
3. Выбери тип **"Game"**
4. Дай имя проекту, например `MySandbox`
5. Нажми **"Create"**

s&box автоматически создаст базовую структуру папок.

## Шаг 1.2: Структура, которая должна получиться

После создания у тебя будет такая структура:

```
MySandbox/
├── MySandbox.sbproj        ← Главный файл проекта
├── ProjectSettings/        ← Папка настроек
├── Assets/                 ← Папка ресурсов
│   └── scenes/             ← Папка сцен
├── Code/                   ← Папка кода
└── Editor/                 ← Папка расширений редактора
```

## Шаг 1.3: Настройка файла проекта (.sbproj)

Файл `.sbproj` — это JSON-файл. Это «паспорт» твоего проекта. Движок читает его первым, чтобы понять, что за проект перед ним.

Открой `MySandbox.sbproj` и приведи его к такому виду:

```json
{
  "Title": "MySandbox",
  "Type": "game",
  "Org": "твой_ник",
  "Ident": "mysandbox",
  "Schema": 1,
  "IncludeSourceFiles": false,
  "Resources": "/fonts/*\n/UI/*.png\n/ui/*.png\n/ui/*",
  "PackageReferences": [],
  "EditorReferences": null,
  "Mounts": [],
  "IsStandaloneOnly": false,
  "Metadata": {
    "MaxPlayers": 32,
    "MinPlayers": 1,
    "TickRate": 50,
    "GameNetworkType": "Multiplayer",
    "MapSelect": "Unrestricted",
    "MapList": [
      "facepunch.flatgrass"
    ],
    "RankType": "None",
    "PerMapRanking": false,
    "LeaderboardType": "None",
    "CsProjName": "",
    "StartupScene": "scenes/sandbox.scene",
    "ControlModes": {
      "Keyboard": true,
      "VR": false,
      "Gamepad": true
    },
    "LaunchMode": "Launcher",
    "MapStartupScene": "scenes/sandbox.scene",
    "DedicatedServerStartupScene": "scenes/sandbox.scene",
    "SystemScene": "scenes/system.scene"
  }
}
```

### Разбор каждого поля:

| Поле | Что значит | Почему такое значение |
|------|-----------|----------------------|
| `Title` | Название игры, видимое игрокам | Имя твоего проекта |
| `Type` | Тип проекта | `"game"` — это полноценная игра |
| `Org` | Организация/автор | Твой никнейм в s&box |
| `Ident` | Уникальный идентификатор | Латиницей, без пробелов |
| `Schema` | Версия формата файла | Всегда `1` |
| `IncludeSourceFiles` | Включать ли код при публикации | `false` — код компилируется, исходники не нужны |
| `Resources` | Какие файлы включать в сборку | Шрифты, картинки для UI |
| `MaxPlayers` | Максимум игроков | `32` — как в оригинальном Sandbox |
| `TickRate` | Частота обновления сервера | `50` — 50 обновлений в секунду |
| `GameNetworkType` | Тип сети | `"Multiplayer"` — мультиплеер |
| `MapSelect` | Выбор карт | `"Unrestricted"` — любые карты |
| `MapList` | Карта по умолчанию | `facepunch.flatgrass` — плоская зелёная карта |
| `StartupScene` | Главная сцена | Путь к нашей сцене |
| `SystemScene` | Системная сцена | Для UI и глобальных систем |

### Как это работает в движке:

Когда ты нажимаешь клавишу `W`, движок s&box переводит это в действие `"Forward"`. В коде ты проверяешь не конкретную клавишу, а действие:

```csharp
// Так НЕ надо:
if ( клавиша W нажата ) двигаться вперёд;

// Так НАДО:
if ( Input.Down( "Forward" ) ) двигаться вперёд;
```

Это позволяет игроку переназначить клавиши, а твой код продолжит работать.

## Шаг 1.5: Настройка физики (Physics.config)

Создай файл `ProjectSettings/Physics.config`:

```json
{
  "UseFixedUpdate": true,
  "SubSteps": 4,
  "FixedUpdateFrequency": 50,
  "MaxFixedUpdates": 2,
  "__guid": "327be197-6281-4b27-9b7f-5f406849093a",
  "__schema": "configdata",
  "__type": "PhysicsSettings",
  "__version": 1
}
```

### Разбор:

| Поле | Что значит |
|------|-----------|
| `UseFixedUpdate` | Использовать фиксированный шаг физики (обязательно `true` для стабильной физики) |
| `SubSteps` | Подшаги физики за один кадр. `4` — хороший баланс между точностью и производительностью |
| `FixedUpdateFrequency` | Частота обновления физики в герцах. `50` — совпадает с `TickRate` сервера |
| `MaxFixedUpdates` | Максимум физических обновлений за один кадр. Защита от «спирали смерти» при лагах |

### Как это работает внутри:

Физический движок обновляется **отдельно** от отрисовки кадров. Если игра работает при 144 FPS, физика всё равно считается 50 раз в секунду. Это нужно для:
1. **Стабильности** — физика не зависит от FPS
2. **Сетевой синхронизации** — все игроки считают физику с одинаковой частотой
3. **Предсказуемости** — верёвки, пружины и суставы ведут себя одинаково


### Как работает система столкновений:

Каждый объект в игре имеет **тег** (метку), который определяет его тип. Таблица столкновений говорит движку, как два объекта с разными тегами должны взаимодействовать:

- **Collide** — столкновение (мяч отскакивает от стены)
- **Trigger** — проход насквозь, но с уведомлением (вошёл в зону)
- **Ignore** — полностью игнорировать друг друга

```
Пример: Пуля (bullet) и Вода (water)
Результат: Ignore
Значит: Пуля пролетает сквозь воду
```

## Шаг 1.7: Настройка сети (Networking.config)

```json
{
  "DestroyLobbyWhenHostLeaves": true,
  "AutoSwitchToBestHost": false,
  "ClientsCanSpawnObjects": false,
  "ClientsCanRefreshObjects": false,
  "UpdateRate": 30,
  "__guid": "bf156628-1b7e-4af7-a101-1d08e8ca36c5",
  "__schema": "configdata",
  "__type": "NetworkingSettings",
  "__version": 1
}
```

### Разбор:

| Поле | Что значит | Зачем |
|------|-----------|-------|
| `DestroyLobbyWhenHostLeaves` | Закрыть сервер, когда хост уходит | `true` — без хоста игра не работает |
| `AutoSwitchToBestHost` | Автоматически менять хоста | `false` — мы сами управляем этим |
| `ClientsCanSpawnObjects` | Могут ли клиенты создавать объекты | `false` — только хост создаёт объекты (безопасность) |
| `ClientsCanRefreshObjects` | Могут ли клиенты обновлять объекты | `false` — защита от читов |
| `UpdateRate` | Частота отправки данных по сети | `30` пакетов в секунду |

> ⚠️ **Важно**: `ClientsCanSpawnObjects: false` — это ключевая настройка безопасности. Все объекты создаются только на сервере (хосте) через RPC-вызовы. Иначе читеры могли бы спавнить что угодно.

## Шаг 1.9: Создание Assembly.cs

Создай файл `Code/Assembly.cs`:

```csharp
global using System;
global using Sandbox;
global using System.Linq;
global using System.Threading.Tasks;
global using System.Collections.Generic;
global using Sandbox.Diagnostics;
```

### Что это?

Это **глобальные using-директивы**. Они говорят компилятору: «В каждом файле проекта автоматически подключай эти библиотеки». Без этого тебе пришлось бы в каждом файле писать:

```csharp
using System;
using Sandbox;
// и т.д.
```

| Библиотека | Что даёт |
|-----------|---------|
| `System` | Базовые типы (string, int, float, Math и т.д.) |
| `Sandbox` | Все классы движка s&box (Component, GameObject, Scene...) |
| `System.Linq` | Удобная работа со списками (`.Where()`, `.Select()`, `.First()`) |
| `System.Threading.Tasks` | Асинхронные операции (загрузка, ожидание) |
| `System.Collections.Generic` | Списки, словари, коллекции (`List<T>`, `Dictionary<K,V>`) |
| `Sandbox.Diagnostics` | Инструменты отладки |
