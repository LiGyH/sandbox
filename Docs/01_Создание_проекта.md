# Этап 1: Создание скелета проекта — ВСЕ конфигурационные файлы

## Цель этого этапа

Создать **полный скелет** проекта s&box Sandbox — со всеми конфигурационными файлами, настройками ввода, физики, столкновений, звука, сети, а также служебными файлами Git и редактора кода. После этого этапа ты сможешь открыть проект в редакторе s&box — он скомпилируется без ошибок, хотя игровой логики пока не будет.

Мы будем создавать точную копию оригинального проекта Sandbox от Facepunch. Каждая строчка каждого файла приведена в точности — ничего не пропущено, ничего не упрощено.

---

## Шаг 1.1: Создание проекта

1. Открой **s&box** через Steam
2. В главном меню нажми **Create a Project** (Создать проект)
3. Выбери тип **Game**
4. Назови проект `Sandbox`
5. Нажми **Create**

s&box создаст базовую структуру папок. Но нам нужно привести **все** файлы к правильному виду. Начнём с самого главного — манифеста проекта.

---

## Шаг 1.2: Манифест проекта — `sandbox.sbproj`

Файл `.sbproj` (от «s&box project») — это **JSON-файл**, который является «паспортом» твоего проекта. Движок s&box читает его самым первым, чтобы понять: что это за проект, кто его автор, какие у него настройки, какую сцену загружать при старте.

Открой файл `sandbox.sbproj` в корне проекта и приведи его к **точно** такому виду:

```json
{
  "Title": "Sandbox",
  "Type": "game",
  "Org": "facepunch",
  "Ident": "sandbox",
  "Schema": 1,
  "IncludeSourceFiles": false,
  "Resources": "/fonts/*\n/UI/*.png\n/ui/*.png\n/ui/*\n/ui/weapons/melee/*",
  "PackageReferences": [],
  "EditorReferences": null,
  "Mounts": [],
  "IsStandaloneOnly": false,
  "Metadata": {
    "MaxPlayers": 32,
    "MinPlayers": 1,
    "TickRate": 50,
    "GameNetworkType": "Multiplayer",
    "MapSelect": "Unrestricted",
    "MapList": [
      "facepunch.flatgrass"
    ],
    "RankType": "None",
    "PerMapRanking": false,
    "LeaderboardType": "None",
    "CsProjName": "",
    "StartupScene": "scenes/sandbox.scene",
    "ControlModes": {
      "Keyboard": true,
      "VR": false,
      "Gamepad": true
    },
    "LaunchMode": "Launcher",
    "MapStartupScene": "scenes/sandbox.scene",
    "DedicatedServerStartupScene": "scenes/sandbox.scene",
    "SystemScene": "scenes/system.scene"
  }
}
```

### Подробный разбор каждого поля

#### Основные поля верхнего уровня

| Поле | Значение | Подробное объяснение |
|------|----------|---------------------|
| `"Title"` | `"Sandbox"` | **Название игры**, которое видят игроки в меню s&box, в списке серверов и в библиотеке. Это человекочитаемое имя — может содержать пробелы, заглавные буквы и т.д. |
| `"Type"` | `"game"` | **Тип проекта**. В s&box есть несколько типов: `"game"` — полноценная игра, `"library"` — переиспользуемая библиотека, `"addon"` — дополнение. Мы делаем игру, поэтому `"game"`. |
| `"Org"` | `"facepunch"` | **Организация или автор**. Это твой никнейм или имя организации в s&box. Используется для формирования уникального идентификатора пакета в формате `org.ident` (т.е. `facepunch.sandbox`). Для своего проекта подставь свой ник. |
| `"Ident"` | `"sandbox"` | **Идентификатор проекта**. Только латинские буквы в нижнем регистре, без пробелов. Вместе с `Org` формирует уникальное имя пакета: `facepunch.sandbox`. |
| `"Schema"` | `1` | **Версия формата** файла `.sbproj`. Всегда `1` — движок использует это, чтобы понимать структуру файла. Не меняй. |
| `"IncludeSourceFiles"` | `false` | **Включать ли исходный код** при публикации пакета. `false` означает, что при публикации в s&box asset store будет включён только скомпилированный код (DLL), а исходники `.cs` — нет. Это защищает твой код от копирования. |
| `"Resources"` | `"/fonts/*\n/UI/*.png\n/ui/*.png\n/ui/*\n/ui/weapons/melee/*"` | **Список путей к ресурсам**, которые включаются в сборку. Символ `\n` — это разделитель строк. Здесь мы включаем: все шрифты (`/fonts/*`), PNG-картинки для интерфейса (`/UI/*.png`, `/ui/*.png`), все файлы UI (`/ui/*`), и отдельно файлы для иконок оружия ближнего боя (`/ui/weapons/melee/*`). Пути `/UI/` и `/ui/` указаны оба из-за возможных различий в регистре на разных ОС. |
| `"PackageReferences"` | `[]` | **Зависимости от других пакетов**. Пустой массив — наш Sandbox не зависит ни от каких внешних пакетов. Если бы мы использовали чужую библиотеку (например, пакет с оружием), мы бы добавили его сюда. |
| `"EditorReferences"` | `null` | **Зависимости только для редактора**. `null` означает, что нет никаких дополнительных библиотек для инструментов редактора. |
| `"Mounts"` | `[]` | **Примонтированные пакеты** (дополнительные ресурсы из других проектов). Пустой массив — мы не монтируем ничего чужого. |
| `"IsStandaloneOnly"` | `false` | **Только автономный режим**. `false` означает, что игра может работать и внутри s&box-лаунчера, и как отдельное приложение. |

#### Поля внутри `Metadata`

Секция `Metadata` содержит настройки, специфичные для типа проекта `"game"`:

| Поле | Значение | Подробное объяснение |
|------|----------|---------------------|
| `"MaxPlayers"` | `32` | **Максимальное количество игроков** на сервере. 32 — стандарт для Sandbox, как в Garry's Mod. Движок использует это для выделения слотов и отображения в списке серверов. |
| `"MinPlayers"` | `1` | **Минимальное количество игроков** для начала игры. `1` — можно играть в одиночку. |
| `"TickRate"` | `50` | **Частота обновления сервера** (тиков в секунду). Сервер обновляет игровое состояние 50 раз в секунду. Это определяет точность физики, попаданий и всей сетевой синхронизации. Значение `50` — это баланс: достаточно точно и не слишком нагружает сервер. Для сравнения: Counter-Strike использует 64 или 128 тиков. |
| `"GameNetworkType"` | `"Multiplayer"` | **Тип сети**. `"Multiplayer"` — многопользовательская игра. Другие варианты: `"Singleplayer"` (только одиночная), `"Both"` (и то, и другое). |
| `"MapSelect"` | `"Unrestricted"` | **Политика выбора карт**. `"Unrestricted"` — сервер может загрузить **любую** карту из мастерской. `"Tagged"` — только карты с определённым тегом, `"Official"` — только из списка `MapList`. Sandbox — свободная песочница, поэтому ограничений нет. |
| `"MapList"` | `["facepunch.flatgrass"]` | **Список карт по умолчанию**. `facepunch.flatgrass` — плоская зелёная карта, идеальная для тестирования и свободной игры. Это стандартная карта Sandbox. |
| `"RankType"` | `"None"` | **Тип рейтинговой системы**. `"None"` — нет рейтинга. Sandbox — песочница, тут нечего ранжировать. |
| `"PerMapRanking"` | `false` | **Отдельный рейтинг для каждой карты**. `false` — не используется (рейтинга и так нет). |
| `"LeaderboardType"` | `"None"` | **Тип таблицы лидеров**. `"None"` — нет таблицы лидеров. |
| `"CsProjName"` | `""` | **Имя C#-проекта**. Пустая строка — движок сгенерирует имя автоматически. |
| `"StartupScene"` | `"scenes/sandbox.scene"` | **Главная сцена игры**. Это сцена, которая загружается при запуске игры. Путь относительный — от папки `Assets/`. |
| `"ControlModes"` | `{ "Keyboard": true, "VR": false, "Gamepad": true }` | **Поддерживаемые устройства ввода**. Клавиатура+мышь — да, VR — нет (Sandbox не VR-игра), геймпад — да. Это влияет на то, какие настройки ввода доступны игроку и какие подсказки по кнопкам показываются в UI. |
| `"LaunchMode"` | `"Launcher"` | **Режим запуска**. `"Launcher"` — игра запускается через лаунчер s&box (стандартный способ). Другой вариант — `"Standalone"` для автономных сборок. |
| `"MapStartupScene"` | `"scenes/sandbox.scene"` | **Сцена при загрузке карты**. Какую сцену загружать, когда сервер переключает карту. Обычно совпадает с `StartupScene`. |
| `"DedicatedServerStartupScene"` | `"scenes/sandbox.scene"` | **Сцена для выделенного сервера**. Какую сцену загружать при запуске выделенного сервера (без графики, только серверная логика). |
| `"SystemScene"` | `"scenes/system.scene"` | **Системная сцена**. Загружается поверх основной сцены, содержит глобальные компоненты: главное меню, настройки, чат, HUD-элементы, которые не зависят от конкретной карты. |

## Шаг 1.3: Настройки редактора кода — `.editorconfig`

Файл `.editorconfig` — это **стандарт индустрии** для настройки форматирования кода. Его понимают практически все редакторы: Visual Studio, VS Code, Rider, и т.д. Когда ты (или кто-то другой) открываешь файл проекта, редактор автоматически применяет правила из `.editorconfig` — какие отступы использовать, где ставить скобки, как форматировать код.

Это критически важно в командной работе: без `.editorconfig` один программист использовал бы табуляции, другой — пробелы, третий — ставил бы скобки на той же строке. Код превратился бы в кашу, а каждый коммит в Git содержал бы сотни бессмысленных изменений форматирования.

Создай файл `.editorconfig` в **корне** проекта (рядом с `sandbox.sbproj`):

### Разбор ключевых настроек

**`root = true`** — это корневой `.editorconfig`. Редактор не будет искать другие `.editorconfig` в родительских папках. Все правила определяются здесь.

**`[*.{cs,razor}]`** — все следующие правила применяются только к файлам C# (`.cs`) и Razor (`.razor`).

**Почему табуляции (`indent_style = tab`), а не пробелы?**
Это стиль Facepunch. Табуляции занимают 1 символ вместо 4, и каждый разработчик может настроить ширину табуляции под себя (кто-то любит 2, кто-то 4, кто-то 8). Пробелы фиксированы — если тебе не нравится ширина, терпи.

**Ключевые правила форматирования:**

| Правило | Значение | Что это значит на практике |
|---------|----------|---------------------------|
| `csharp_new_line_before_open_brace = all` | Скобка на новой строке | Стиль Allman: `if (x)\n{` вместо `if (x) {` |
| `csharp_space_between_method_call_parameter_list_parentheses = true` | Пробелы внутри скобок вызова | `DoSomething( x, y )` вместо `DoSomething(x, y)` |
| `csharp_space_between_method_declaration_parameter_list_parentheses = true` | Пробелы внутри скобок объявления | `void Foo( int x )` вместо `void Foo(int x)` |
| `csharp_space_between_parentheses = control_flow_statements` | Пробелы в управляющих конструкциях | `if ( condition )` вместо `if (condition)` |
| `csharp_style_expression_bodied_methods = false` | Методы — без `=>` | Полные блоки `{ }` для методов (лучше читаются) |
| `csharp_style_expression_bodied_properties = true` | Свойства — через `=>` | `public int X => _x;` (коротко и ясно) |

> 💡 Обрати внимание: в s&box-проектах принято **ставить пробелы** внутри круглых скобок: `if ( x > 0 )` и `DoThing( arg )`. Это необычный стиль — во многих других C#-проектах пробелов нет. Но здесь мы следуем стилю Facepunch.

## Шаг 1.4: Файл игнорирования Git — `.gitignore`

Файл `.gitignore` говорит системе контроля версий Git: «Не отслеживай эти файлы». Это нужно, чтобы в репозиторий не попадали временные файлы, скомпилированные бинарники, настройки конкретного разработчика и автоматически сгенерированные файлы.

Создай файл `.gitignore` в корне проекта:

```
# This file describes files and paths that should not be tracked by Git version control
# https://git-scm.com/docs/gitignore

# Auto-generated code editor files
.vs/*
.vscode/*
*.csproj
obj
bin
Properties/*
code/obj/*
code/Properties/*

# Auto-generated asset related files
.sbox/*
*.generated.*
*.*_c
!*.shader_c
*.los
*.vpk
*launchSettings.json
*.sln

*idea

# Exported / standalone games
Exports/
*.user
*.slnx
```

### Разбор каждой секции

**Файлы редакторов:**
- `.vs/*` — папка настроек Visual Studio (у каждого разработчика своя)
- `.vscode/*` — папка настроек VS Code
- `*.csproj` — файл C#-проекта (генерируется автоматически движком s&box из `.sbproj`)
- `obj`, `bin` — папки компиляции (промежуточные и конечные файлы)
- `Properties/*` — автоматически генерируемые свойства проекта
- `code/obj/*`, `code/Properties/*` — то же самое для подпапки `code`

**Файлы движка и ассетов:**
- `.sbox/*` — внутренний кэш движка s&box
- `*.generated.*` — автоматически сгенерированные файлы
- `*.*_c` — скомпилированные ресурсы (модели, текстуры и т.д.). Символ `_c` означает «compiled»
- `!*.shader_c` — **исключение**: скомпилированные шейдеры **нужно** хранить (знак `!` отменяет предыдущее правило)
- `*.los` — файлы Line of Sight (данные видимости)
- `*.vpk` — пакеты Valve (архивы ресурсов)
- `*launchSettings.json` — настройки запуска (индивидуальны)
- `*.sln` — файл решения Visual Studio (генерируется автоматически)
- `*idea` — папка настроек JetBrains Rider/IntelliJ

**Экспорт:**
- `Exports/` — папка экспортированных (скомпилированных) сборок игры
- `*.user` — пользовательские настройки Visual Studio
- `*.slnx` — формат файла решения Visual Studio нового поколения

## Шаг 1.5: Атрибуты Git — `.gitattributes`

Файл `.gitattributes` управляет тем, как Git обрабатывает файлы — в частности, окончания строк.

Создай файл `.gitattributes` в корне проекта:

```
# Auto detect text files and perform LF normalization
* text=auto
```

### Что это делает?

`* text=auto` — для **всех** файлов (`*`) Git автоматически определяет, текстовый это файл или бинарный. Для текстовых файлов Git нормализует окончания строк:
- В репозитории хранятся с LF (Unix-стиль: `\n`)
- При извлечении на Windows автоматически преобразуются в CRLF (`\r\n`)
- При извлечении на Mac/Linux остаются LF

Это решает классическую проблему: когда один разработчик работает на Windows, а другой — на Mac, без этой настройки Git показывал бы изменения во **всех** строках каждого файла, даже если изменилось только окончание строки.

## Шаг 1.6: Настройка столкновений — `ProjectSettings/Collision.config`

Файл столкновений определяет **физическое взаимодействие** между объектами. В s&box каждый физический объект имеет **слой столкновений** (collision layer) — тег, описывающий его тип. Таблица столкновений говорит движку, как объекты с разными тегами должны реагировать друг на друга.

Создай файл `ProjectSettings/Collision.config`:

```json
{
  "Version": 2,
  "Defaults": {
    "solid": "Collide",
    "trigger": "Trigger",
    "ladder": "Ignore",
    "water": "Trigger",
    "ragdoll": "Ignore",
    "player": "Unset",
    "effect": "Ignore",
    "bullet": "Unset",
    "bolt": "Collide",
    "projectile": "Collide",
    "sound": "Ignore",
    "noclip": "Ignore",
    "constraint": "Collide"
  },
  "Pairs": [
    {
      "a": "solid",
      "b": "solid",
      "r": "Collide"
    },
    {
      "a": "trigger",
      "b": "playerclip",
      "r": "Ignore"
    },
    {
      "a": "trigger",
      "b": "solid",
      "r": "Trigger"
    },
    {
      "a": "playerclip",
      "b": "solid",
      "r": "Collide"
```

### Что такое слои столкновений?

Представь, что каждый объект в игре носит «бейджик» с надписью — это его слой. Когда два объекта сближаются, движок смотрит на их бейджики и решает, что делать.

#### Все слои и их назначение:

| Слой | Что это | Примеры объектов |
|------|---------|-----------------|
| `solid` | Твёрдые объекты | Стены, пол, пропы (мебель, бочки) |
| `trigger` | Триггерные зоны | Невидимые зоны: зона смерти, зона телепорта |
| `ladder` | Лестницы | Вертикальные лестницы, по которым можно лазить |
| `water` | Вода | Водные поверхности и объёмы |
| `ragdoll` | Тряпичные куклы | Тела убитых персонажей (ragdoll-физика) |
| `player` | Игрок | Коллайдер живого игрока |
| `effect` | Визуальные эффекты | Частицы, декали, эффекты взрывов |
| `bullet` | Пули | Мгновенные пули (hitscan — без физического объекта) |
| `bolt` | Снаряды | Физические снаряды (арбалет, гранаты) |
| `projectile` | Проектайлы | Общий слой для летящих объектов |
| `sound` | Звук | Невидимые объекты для распространения звука |
| `noclip` | Noclip-режим | Игрок в режиме полёта сквозь стены |
| `constraint` | Связи | Верёвки, пружины, шарниры между объектами |
| `playerclip` | Невидимые стены | Невидимые барьеры, блокирующие только игрока |

#### Три типа взаимодействия:

| Тип | Что происходит | Пример |
|-----|---------------|--------|
| **Collide** | Объекты **сталкиваются** физически | Мяч отскакивает от стены |
| **Trigger** | Объекты **проходят друг через друга**, но движок **уведомляет** код | Игрок входит в триггерную зону — срабатывает событие |
| **Ignore** | Объекты **полностью игнорируют** друг друга | Пуля пролетает сквозь воду |

#### Примеры из таблицы `Pairs`:

```
solid + solid = Collide      → Ящик врезается в стену — отскакивает
trigger + solid = Trigger     → Игрок входит в триггер — получаем событие
water + bullet = Ignore       → Пуля пролетает сквозь воду
noclip + * = Ignore           → Игрок в нoclip пролетает сквозь всё
ragdoll + bullet = Collide    → Пуля попадает в труп — труп дёргается
player + bolt = Trigger       → Снаряд попадает в игрока — уведомление (для нанесения урона)
constraint + constraint = Ignore → Верёвки не сталкиваются друг с другом
```

> 💡 Обрати внимание на слой `noclip`: он имеет `Ignore` с **каждым** другим слоем. Это логично — режим noclip позволяет летать сквозь стены, поэтому физика не должна его блокировать.

Поля `__guid`, `__schema`, `__type`, `__version` — это служебные метаданные движка s&box. Они нужны для сериализации и не должны изменяться вручную.

## Шаг 1.7: Настройка ввода — `ProjectSettings/Input.config`

Файл `Input.config` определяет **все** игровые действия и их привязки к клавишам клавиатуры и кнопкам геймпада. Это полный набор действий Sandbox — от базового перемещения до меню спавна и отмены действий.

Суть системы ввода s&box: ты **никогда** не проверяешь конкретные клавиши в коде. Вместо этого ты проверяешь **действия**. Это позволяет игрокам переназначать клавиши, а также поддерживать разные устройства (клавиатуру, геймпад).

```csharp
// ❌ Плохо — привязка к конкретной клавише:
if ( Keyboard.IsKeyDown( Key.W ) ) МовеФорвард();

// ✅ Хорошо — привязка к действию:
if ( Input.Down( "Forward" ) ) MoveForward();
```

```
│       └── ThirdPerson (100%) ← Звуки оружия от третьего лица (чужое оружие)
│
└── UI (75%) ← Звуки интерфейса (чуть тише, чтобы не отвлекать от игры)
    Spacializing: 0    — не пространственные
    DistanceAttenuation: 0 — не зависят от расстояния
```

### Разбор ключевых параметров микшера

| Параметр | Что значит | Значения |
|----------|-----------|----------|
| `Volume` | Громкость канала | `0.0` (тишина) — `1.0` (полная громкость). У Weapons стоит `0.25` — только 25% громкости, потому что выстрелы очень громкие |
| `Mute` | Заглушить канал | `true`/`false`. Полезно для отладки — заглушить все звуки оружия |
| `Solo` | Соло (только этот канал) | `true`/`false`. Включив Solo на Music, ты услышишь только музыку |
| `Spacializing` | Пространственность | `0` — 2D-звук (одинаков в обоих колонках). `1` — полный 3D (зависит от направления) |
| `MaxVoices` | Максимум одновременных звуков | `64` — сколько звуков может играть одновременно в этом канале |
| `DistanceAttenuation` | Затухание с расстоянием | `0` — не затухает (музыка). `1` — полное затухание (далёкий выстрел тихий) |
| `Occlusion` | Приглушение за стенами | `0` — стены не влияют. `1` — полное приглушение за стенами |
| `AirAbsorption` | Поглощение воздухом | `0` — нет. `1` — далёкие звуки теряют высокие частоты (как в реальности) |
| `OverrideOcclusion` | Переопределить окклюзию | `false` — использовать стандартную. `true` — собственные правила по тегам |
| `Processors` | Аудиоэффекты | Массив обработчиков (реверберация, эхо). Пустой — без эффектов |
| `IsDefault` | Канал по умолчанию | Звуки без явно указанного микшера попадают в этот канал. У `Game` стоит `true` |

> 💡 **Почему Weapons на 25%?** Звуки выстрелов обычно записываются на максимальной громкости. Без уменьшения они перекрыли бы все остальные звуки. 25% — это компромисс: выстрелы слышно, но они не заглушают шаги и голос.

> 💡 **Зачем разделять FirstPerson и ThirdPerson?** Звук **твоего** выстрела (от первого лица) должен звучать иначе, чем звук **чужого** выстрела (от третьего лица). Твой выстрел — громкий, хлёсткий, «прямо у уха». Чужой — приглушённый, пространственный, с эхом. Разные микшеры позволяют настроить это.

## Шаг 1.9: Настройка сети — `ProjectSettings/Networking.config`

Создай файл `ProjectSettings/Networking.config`:

```json
{
  "DestroyLobbyWhenHostLeaves": true,
  "AutoSwitchToBestHost": false,
  "ClientsCanSpawnObjects": false,
  "ClientsCanRefreshObjects": false,
  "UpdateRate": 30,
  "__guid": "bf156628-1b7e-4af7-a101-1d08e8ca36c5",
  "__schema": "configdata",
  "__type": "NetworkingSettings",
  "__version": 1
}
```

### Разбор каждого поля

| Поле | Значение | Подробное объяснение |
|------|----------|---------------------|
| `DestroyLobbyWhenHostLeaves` | `true` | Когда хост (создатель сервера) выходит из игры, лобби **уничтожается** и все игроки отключаются. Это стандартное поведение для listen-серверов (когда один игрок является и сервером). |
| `AutoSwitchToBestHost` | `false` | **Не** переключать хоста автоматически. Если бы было `true`, при выходе хоста движок попытался бы назначить другого игрока хостом. Мы отключаем это для простоты и предсказуемости. |
| `ClientsCanSpawnObjects` | `false` | **Клиенты НЕ могут** создавать сетевые объекты напрямую. Это **ключевая настройка безопасности**. Все объекты создаются только на сервере через RPC-вызовы (Remote Procedure Call). Без этого ограничения читер мог бы создать 1000 бочек и завалить сервер. |
| `ClientsCanRefreshObjects` | `false` | **Клиенты НЕ могут** обновлять свойства сетевых объектов напрямую. Опять безопасность: изменения состояния идут только через сервер. |
| `UpdateRate` | `30` | **Частота отправки сетевых обновлений** — 30 пакетов в секунду. Это значит, что каждые ~33 миллисекунды сервер отправляет клиентам обновлённое состояние мира. Между обновлениями клиент **интерполирует** (плавно достраивает) позиции объектов. 30 — хороший баланс между плавностью и нагрузкой на сеть. |

> ⚠️ **Важно**: `ClientsCanSpawnObjects: false` и `ClientsCanRefreshObjects: false` — это фундамент сетевой безопасности. Вся логика создания и изменения объектов происходит на сервере. Клиент может только **попросить** сервер что-то сделать (через `[Rpc.Broadcast]`), а сервер решает, разрешить это или нет.

## Шаг 1.10: Настройка физики — `ProjectSettings/Physics.config`

Создай файл `ProjectSettings/Physics.config`:

```json
{
  "UseFixedUpdate": true,
  "SubSteps": 4,
  "FixedUpdateFrequency": 50,
  "MaxFixedUpdates": 2,
  "__guid": "327be197-6281-4b27-9b7f-5f406849093a",
  "__schema": "configdata",
  "__type": "PhysicsSettings",
  "__version": 1
}
```

### Разбор каждого поля

| Поле | Значение | Подробное объяснение |
|------|----------|---------------------|
| `UseFixedUpdate` | `true` | **Использовать фиксированный шаг физики**. Обязательно `true` для стабильной физики. Без этого физика обновлялась бы с переменной частотой (зависящей от FPS), и объекты вели бы себя по-разному на разных компьютерах. |
| `SubSteps` | `4` | **Количество подшагов** физики за один основной шаг. Чем больше подшагов, тем точнее физика (особенно для быстрых объектов и тонких стен), но тем выше нагрузка на процессор. `4` — хороший баланс: достаточно для корректной работы верёвок, пружин и быстрых снарядов. |
| `FixedUpdateFrequency` | `50` | **Частота обновления физики** в герцах — 50 раз в секунду. Совпадает с `TickRate` сервера (тоже 50). Это важно: физика и сетевая логика работают синхронно, что упрощает предсказание движения и синхронизацию между клиентами. |
| `MaxFixedUpdates` | `2` | **Максимум физических обновлений за один кадр рендера**. Защита от «спирали смерти»: если компьютер тормозит и не успевает обновлять физику, он не пытается «нагнать» бесконечно (что бы ещё больше его замедлило), а пропускает лишние обновления. Максимум 2 шага физики за 1 кадр. |

### Как это работает:

Физический движок обновляется **отдельно** от отрисовки кадров:

```
Монитор 144 Гц: Рендер → 144 кадров/сек
Физика:         50 обновлений/сек (каждые 20 мс)
Каждое обновление физики: 4 подшага → 200 мини-расчётов/сек
```

Это обеспечивает:
1. **Стабильность** — физика не зависит от FPS (60 FPS или 300 FPS — физика одинакова)
2. **Сетевую синхронизацию** — все игроки считают физику с одинаковой частотой (50 Гц)
3. **Предсказуемость** — верёвки, пружины и суставы (constraints) ведут себя одинаково на всех машинах

## Шаг 1.11: Глобальные импорты для кода — `Code/Assembly.cs`

В C# каждый файл обычно начинается с нескольких строк `using ...`, подключающих нужные библиотеки. Но если одни и те же библиотеки нужны в **каждом** файле проекта, это превращается в бессмысленное повторение.

В C# 10 появилась возможность **глобальных using** — ты объявляешь их **один раз** в одном файле, и они автоматически доступны во **всех** файлах проекта.

Создай файл `Code/Assembly.cs`:

```csharp
global using System;
global using Sandbox;
global using System.Linq;
global using System.Threading.Tasks;
global using System.Collections.Generic;
global using Sandbox.Diagnostics;
```

### Подробное объяснение каждого пространства имён

| Пространство имён | Что даёт | Примеры использования |
|-------------------|---------|----------------------|
| `System` | **Базовые типы языка C#** — строки, числа, математика, исключения. Без этого не работает практически ничего. | `string`, `int`, `float`, `Math.Abs()`, `Console.WriteLine()`, `Exception` |
| `Sandbox` | **Все классы движка s&box**. Это сердце проекта — компоненты, игровые объекты, сцены, сеть. Без этого ты не сможешь написать ни одного компонента. | `Component`, `GameObject`, `Scene`, `Input`, `Sound`, `Gizmo`, `[Property]` |
| `System.Linq` | **LINQ** (Language Integrated Query) — мощная система запросов к коллекциям. Позволяет фильтровать, сортировать и преобразовывать списки в одну строку. | `list.Where( x => x.IsAlive )`, `list.Select( x => x.Name )`, `list.First()`, `list.Any()` |
| `System.Threading.Tasks` | **Асинхронное программирование** — работа с задачами, которые выполняются не мгновенно (загрузка ресурсов, сетевые запросы, таймеры). | `Task`, `async`, `await`, `Task.Delay()`, `Task.WhenAll()` |
| `System.Collections.Generic` | **Обобщённые коллекции** — типизированные списки, словари, множества. Используются повсеместно. | `List<GameObject>`, `Dictionary<string, int>`, `HashSet<Player>`, `Queue<Action>` |
| `Sandbox.Diagnostics` | **Инструменты отладки и диагностики** движка — проверки условий (assertions), логирование, метрики производительности. | `Assert.True( health > 0 )`, `Assert.NotNull( player )` |

> 💡 Благодаря этому файлу, когда ты создашь новый файл `Code/MyComponent.cs`, тебе **не нужно** писать `using System;`, `using Sandbox;` и т.д. — они уже подключены глобально.

## Шаг 1.13: Расширения движка — `Code/EngineAdditions.cs`

Этот файл содержит **интерфейс** и **расширение** (extension method), которые дополняют встроенные классы движка s&box. Это первый «настоящий» C#-код нашего проекта.

Создай файл `Code/EngineAdditions.cs`:

```csharp
namespace Sandbox;

public interface IDefinitionResource
{
	public string Title { get; set; }
	public string Description { get; set; }
}

public static class EngineAdditions
{
	extension( GameObject go )
	{
		public GameObject FindNetworkRoot()
		{
			if ( !go.IsValid() ) return null;

			if ( go.NetworkMode == NetworkMode.Object ) return go;
			return go.Parent?.FindNetworkRoot();
		}
	}
}
```

### Подробный разбор

#### Интерфейс `IDefinitionResource`

```csharp
public interface IDefinitionResource
{
	public string Title { get; set; }
	public string Description { get; set; }
}
```

**Что такое интерфейс?** Это «контракт» — он говорит: «Любой класс, реализующий этот интерфейс, **обязан** иметь свойства `Title` и `Description`».

**Зачем нужен `IDefinitionResource`?** В Sandbox есть несколько типов ресурсов, которые пользователь может настраивать: описания сущностей (ScriptedEntity), ресурсы пост-обработки (PostProcessResource), определения инструментов (ToolResource) и т.д. Все они имеют название и описание. Интерфейс `IDefinitionResource` позволяет обращаться с ними **единообразно** — например, показывать список всех ресурсов в UI, не заботясь о конкретном типе.

```csharp
// Пример: показать список ЛЮБЫХ ресурсов с названием и описанием
void ShowResourceList( IEnumerable<IDefinitionResource> resources )
{
    foreach ( var res in resources )
    {
        Log.Info( $"{res.Title}: {res.Description}" );
    }
}
```

#### Расширение `FindNetworkRoot()` для `GameObject`

```csharp
public static class EngineAdditions
{
	extension( GameObject go )
	{
		public GameObject FindNetworkRoot()
		{
			if ( !go.IsValid() ) return null;

			if ( go.NetworkMode == NetworkMode.Object ) return go;
			return go.Parent?.FindNetworkRoot();
		}
	}
}
```

**Что такое extension (расширение)?** Это способ **добавить метод** к существующему классу, не изменяя его исходный код. В данном случае мы добавляем метод `FindNetworkRoot()` к классу `GameObject`, который является частью движка s&box.

> 💡 Синтаксис `extension( GameObject go )` — это синтаксис расширений s&box/C#. Он говорит: «Добавь следующие методы к классу `GameObject`, где `go` — это экземпляр, на котором вызывается метод».

**Что делает `FindNetworkRoot()`?** Он **поднимается вверх по иерархии** объектов (от ребёнка к родителю) и ищет **корневой сетевой объект** — то есть объект с `NetworkMode == NetworkMode.Object`.

Зачем это нужно? В s&box объекты образуют **дерево**: у оружия есть модель, у модели есть дуло, у дула есть точка вылета частиц. Когда ты подбираешь оружие, тебе нужно найти «корень» этой иерархии — объект, который является самостоятельной сетевой сущностью.

```
Player (NetworkMode.Object) ← ЭТО корень (FindNetworkRoot вернёт его)
├── Body
│   ├── Head
│   └── Arms
│       └── Weapon (мы вызываем FindNetworkRoot() здесь)
│           ├── Model
│           └── MuzzleFlash
```

Пошаговая логика:
1. `if ( !go.IsValid() ) return null;` — если объект уничтожен или невалиден, вернуть `null`
2. `if ( go.NetworkMode == NetworkMode.Object ) return go;` — если этот объект — сетевой корень, вернуть его
3. `return go.Parent?.FindNetworkRoot();` — иначе, спросить у родителя (рекурсия). Оператор `?.` — безопасный вызов: если родителя нет (`null`), просто вернуть `null`

> 💡 `namespace Sandbox;` — код размещён в пространстве имён движка `Sandbox`. Это необходимо, чтобы метод расширения (extension) работал на классе `GameObject`, который тоже находится в `Sandbox`.
