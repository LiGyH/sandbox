# 27.08 — Масштабирование: от 16 до 1000+ игроков

## Что мы делаем?

Берём всё из [27.06](27_06_Network_Internals.md) и [27.07](27_07_Networking_Patterns.md) и составляем **6 профилей** dedicated-сервера: на 16, 64, 128, 256, 1000 и 1000+ игроков. Для каждого — настройки `Networking.config`, обязательные паттерны, ловушки и реальные ограничения движка.

> ⚠️ **Все цифры — ориентиры**, не контракт. Реальная ёмкость зависит от: жанра ([27.09](27_09_Genre_Examples.md)), карты, числа сетевых объектов, железа сервера, среднего пинга. Тестируй нагрузку.

## Общие принципы (вне зависимости от числа)

1. **Один сетевой объект ≠ один игрок.** На 16 игроков легко быть 5000 сетевых пропов (Sandbox-режим).
2. **Главный параметр — *количество видимых* объектов на клиента**, а не общее число.
3. **Сначала — visibility ([26.14](26_14_Network_Visibility.md))**, потом — снижение `UpdateRate`.
4. **Меньше `[Sync]`-полей** на горячих объектах — сильнее всего экономит.

## Профиль 1. До 16 игроков (стандарт)

**Кейсы.** Listen-server друзья, маленький deathmatch, co-op, sandbox.

| Параметр | Значение |
|---|---|
| Сервер | listen или dedicated — оба ок |
| `UpdateRate` | 30 (по умолчанию из [00.08](00_08_Настройки_сети.md)) |
| `INetworkVisible` | не нужен (`AlwaysTransmit = true` всем) |
| Prediction / Lag-comp | не нужны (если co-op), нужны (если PvP) |
| Железо сервера | 2 ядра, 4 ГБ, домашний канал |

**Что делать:**
- Просто следуй базовому туториалу из Фаз 0–25.
- Не оптимизируй заранее.

**Что НЕ делать:**
- ❌ Не вкручивай interest management «на будущее» — добавишь, когда понадобится.

## Профиль 2. До 64 игроков (классический шутер / RP-старт)

**Кейсы.** CS-подобный, конкурентный шутер, маленький RP-сервер.

| Параметр | Значение |
|---|---|
| Сервер | **только dedicated** |
| `UpdateRate` | 30–64 (выше — комфортнее, ниже — экономит CPU) |
| `INetworkVisible` | желательно для пропов и трупов; игроки — без |
| Prediction / Reconciliation | **обязательно** ([27.07](27_07_Networking_Patterns.md)) |
| Lag-comp | **обязательно** для шутера |
| Железо сервера | 4 ядра, 8 ГБ, серверный канал ≥ 100 Мбит |

**Что делать:**
- Перевести всю сеть на server-authoritative модель.
- Логировать ping каждые 10 секунд.
- Проверить, что физика стабильно укладывается в `FixedUpdate` (50 Hz) на пиковом числе игроков.

**Что НЕ делать:**
- ❌ Не давай клиентам спавнить объекты (`ClientsCanSpawnObjects = false`).
- ❌ Не используй `[Sync]` для часто меняющихся низкоприоритетных полей (например, «временный шум камеры»).

## Профиль 3. До 128 игроков (Battle Royale, крупный PvP)

**Кейсы.** PUBG-подобное, Squad-подобное, большой RP.

| Параметр | Значение |
|---|---|
| `UpdateRate` | 30 (выше — съест канал) |
| `INetworkVisible` | **обязателен на всех «не-сюжетных» объектах** |
| Area-of-interest | дистанция 200–400 м от игрока |
| Custom snapshot data | желательно для оптимизации траекторий пуль |
| Железо сервера | 8 ядер, 16 ГБ, выделенный сервер |

**Что делать:**
- Делить карту на квадраты («чанки»), внутри — список «локальных» объектов.
- На кадровом уровне `IsVisibleToConnection` вызывать раз в 2–3 tick-а, не каждый.
- Оптимизировать сериализацию: убрать неиспользуемые `[Sync]`-поля.
- Замерять `Connection.Ping` и `Connection.PacketLoss`, репортить в метрики.

**Что НЕ делать:**
- ❌ Не строить «карту в карте» (вложенные сцены) для оптимизации — добавляет сложности больше, чем экономит.
- ❌ Не делать одну гигантскую `INetworkListener` с тяжёлой логикой — она вызывается из главного потока сервера.

## Профиль 4. До 256 игроков (большой RP, симулятор сообщества)

**Кейсы.** Большие RP-сервера (Five-M-уровень), open-world симуляторы.

| Параметр | Значение |
|---|---|
| `UpdateRate` | 20–25 (опускаем для экономии) |
| Visibility | **жёсткий грид** ([27.07](27_07_Networking_Patterns.md), Interest Mgmt) |
| Все `[Sync]`-поля | **аудит каждого**: «реально ли надо всем?» |
| Снижение разрешения данных | передавать `Vector3` через 16-битные компоненты (custom snapshot data) |
| Железо | 12+ ядер, 32 ГБ, выделенный bare-metal |

**Что делать:**
- Завести **Service-слой**: вся серверная логика — в singleton-сервисах ([27.04](27_04_Local_Project_Serverside_Code.md)), не в компонентах сцены.
- Профилировать каждые сутки: размер snapshot-а на «среднего» клиента не должен расти.
- Отдельно мониторить **самые тяжёлые** клиенты: один с пингом 300 мс может «вытянуть» tick-цикл на хвост.

**Что НЕ делать:**
- ❌ Не пытайся сделать всё в одной сцене. Раздели на: hub-сцена + субсцены (другие города/этажи).
- ❌ Не пихай RPC по 30 в секунду на каждого игрока — упрёшься в сериализацию.

## Профиль 5. До 1000 игроков (MMO-light)

**Кейсы.** «Маленький WoW», большой социальный сервер.

> ⚠️ **На этом уровне s&box работает на пределе возможного.** Facepunch официально не заявляет 1000 игроков как «штатный» режим. Готовься к экспериментам.

| Параметр | Значение |
|---|---|
| `UpdateRate` | 10–15 (медленный мир) |
| Архитектура | **shard-ит сервера**: 4 сервера по 250, объединены вашим бэкендом |
| Visibility | агрессивный грид + LOD (далёкие — не двигаются вообще) |
| Persistent data | **внешняя БД** обязательна (PostgreSQL, Redis); см. [27.10](27_10_Hosting_Operations.md) |
| Железо | по серверу: 16 ядер, 64 ГБ; плюс БД-сервер |

**Что делать:**
- **Шардинг.** Один dedicated тянет ≤ 250 игроков комфортно. Для 1000 — 4 параллельных + ваш собственный «бэкенд» для перемещения игроков между ними и общей БД.
- **Кеширование чанков.** Загружаем мир по требованию (стриминг).
- **Микросервисы вне s&box.** Чат, торговля, рейтинги — отдельные процессы, говорящие с серверами через HTTP/WebSocket ([26.12](26_12_Http_Requests.md), [26.13](26_13_WebSockets.md)).
- **Сильный анти-чит.** На 1000 человек найдётся 5 читеров — и они «разгонят» ваш CPU своими RPC.

**Что НЕ делать:**
- ❌ Не пытайся сделать «бесшовный мир» на одном сервере. Это нереально.
- ❌ Не верь, что hotload вечен — на 1000 человек желательно делать обновления через рестарт с миграцией.

## Профиль 6. 1000+ игроков (MMO в полном смысле)

**Кейсы.** Massive — EVE-подобное, RuneScape-подобное.

> 🚨 **s&box не предназначен для этого "из коробки".** Это не значит «нельзя» — но почти всё придётся писать свой стек поверх. Готовь команду из 5+ инженеров и не ставь коммерческий продукт без полугода R&D.

**Архитектура (только идея):**

```
                    ┌─────────────┐
                    │  Login API  │  (HTTP)
                    └──────┬──────┘
                           │
       ┌───────────────────┼───────────────────┐
       │                   │                   │
┌──────▼──────┐    ┌──────▼──────┐    ┌──────▼──────┐
│  Shard 1    │    │  Shard 2    │    │  Shard N    │
│ (sbox-srv)  │    │ (sbox-srv)  │    │ (sbox-srv)  │
│ 250 players │    │ 250 players │    │ 250 players │
└──────┬──────┘    └──────┬──────┘    └──────┬──────┘
       │                  │                  │
       └────────┬─────────┴──────────────────┘
                │
        ┌───────▼────────┐    ┌──────────────┐
        │  Postgres /    │    │   Redis      │
        │  World DB      │    │ (cache, chat)│
        └────────────────┘    └──────────────┘
```

| Что | Где живёт |
|---|---|
| Перемещение между шардами | бэкенд: kick от шарда 1 → connect к шарду 2 |
| Чат | отдельный микросервис (не sbox), клиент держит WebSocket к нему |
| Торговля, банк, аукцион | отдельный сервис, шарды читают/пишут через API |
| Рейтинги, статы | агрегация в воркерах раз в N минут |
| Сцены | каждый шард — свой кусок мира; перетекание = смена шарда |

**Реалистично ожидать:**

- Производительность в 30–60 игроков **в видимой зоне** одного шарда — нормально.
- 250 «подключённых, но размазанных по миру» — реалистично.
- Бесшовное «вижу 1000 человек на одном экране» — **нет**.

**Это не «как сделать MMO», а «как НЕ запретить себе масштабироваться»**. Подробные архитектуры — за рамками этой документации.

## Универсальный чек-лист «до запуска под нагрузку»

1. ⏱ **Профилирование snapshot-а.** Залогируй средний размер snapshot-а на клиента. Цель: ≤ 4–8 КБ/snapshot для шутера, ≤ 16 КБ для RP.
2. 📈 **Мониторинг tick.** `Time.FixedDelta` должен быть стабилен (≈ 0.02 при 50 Hz). Если пляшет — упор в CPU.
3. 📡 **Метрики на каждом этапе** (см. [27.10](27_10_Hosting_Operations.md)): online, ping, packetloss, RAM, CPU, GC pauses.
4. 🧪 **Стресс-тест ботами.** Напиши тестовых ботов (`Component`, имитация игрока через локальный API), запусти 100 копий, наблюдай.
5. 🛑 **Graceful degradation.** Когда CPU > 80%, **отключай** косметику (визуальные эффекты, далёкие звуки), а не игровую логику.

## Что важно запомнить

- 16 игроков — почти ничего не делать. 64 — все базовые паттерны. 128–256 — visibility и грид. 1000+ — шардинг и внешний бэкенд.
- Главный рычаг — `INetworkVisible` ([26.14](26_14_Network_Visibility.md)).
- `UpdateRate` снижай **после** того, как visibility отжата.
- s&box без архитектурного шардинга комфортно тянет до **~250 игроков на сервер**, дальше — нужны хитрости.
- Все цифры — ориентиры; всегда нагрузочно тестируй на ботах.
