# 00.17 — Hotloading (живая перезагрузка кода)

## Что мы делаем?

Разбираемся в **hotloading** — одной из главных «фишек» s&box. Ты сохраняешь `.cs` или `.razor` файл — движок **автоматически перекомпилирует** и подхватывает изменения, не перезапуская игру и не теряя состояние сцены. Правильно этим пользоваться = кратно ускорить разработку.

## Как это работает

1. Ты нажимаешь **Save** в файле.
2. s&box компилирует изменившуюся сборку.
3. Движок «шагает по памяти» и находит все экземпляры изменённых типов.
4. Обновляет их «на лету» — новая версия класса заменяет старую, **данные в полях сохраняются**.
5. Движок продолжает работать с новой версией кода.

В 99% случаев ты этого даже не замечаешь. Игрок стоит там, где стоял, пуля летит дальше, UI не моргает.

## IL Hotload (быстрый путь)

Если ты поменял **только тело метода** (не сигнатуру, не поля, не новые типы), движок использует **IL Hotload** — просто подменяет инструкции метода **без** прохода по памяти. Это почти мгновенно.

Включено по умолчанию, отключается в Editor Settings → General.

## Что делать, если что-то сломалось

### 1. Посмотри вывод в консоли

При hotload в консоли выводятся предупреждения, если что-то не удалось обновить: лямбды, делегаты, generic-типы и прочее.

### 2. Команда `hotload_log 2`

Введи в консоли, сделай hotload — получишь подробную таблицу: какие типы обрабатывались, сколько это заняло, сколько инстансов. Если видишь «тысячи инстансов» от класса, которого должно быть один — это **утечка**.

### 3. Если совсем плохо — **Restart Editor**

Hotload не боги, он не обязан справляться со всем. Переименовал класс, сильно изменил иерархию типов, сменил generic-параметры — иногда проще перезапустить редактор.

## Что hotload НЕ любит

### Переименование / удаление типа

Hotload не знает, что `OldName` = `NewName`. Он решит, что `OldName` **удалён**, и все ссылки на него станут `null`. Компоненты такого типа удалятся. Если переименовываешь много — перезапускай редактор после.

### Изменение дефолтных значений полей

```csharp
// Было
public static string Greeting = "Hello";

// Стало — но hotload ОСТАВИТ "Hello", не "World"
public static string Greeting = "World";
```

Потому что движок копирует значения старых полей в новые. **Обходится:**
- `const` — применится сразу (`const string X = "...";`).
- Property с get-выражением: `public static string Greeting => "World";` — применится.
- `[SkipHotload]` — скажет hotload, «не копируй это поле».

### Лямбды и делегаты

Если ты **переупорядочил** лямбды внутри метода, или сильно поменял их тела, подписки на события могут «сломаться» — hotload подставит заглушку, которая будет логировать предупреждение. Чинится переподпиской или рестартом.

### Static поля в generic-типах

```csharp
public class Cache<T>
{
    public static Dictionary<string, T> Store = new();  // ⚠️
}
```

Такие поля **забываются** при hotload. Заведи поле только если обязательно — и пометь `[SkipHotload]`, чтобы подавить предупреждение.

### Кеши рефлексии

Если ты кешируешь результаты `GetMethods()`, `GetProperties()` и т.п., после hotload в кеше могут лежать ссылки на **старые** типы. Помечай кеш `[SkipHotload]` и **сбрасывай его** в hotload-коллбеке.

### Dictionary / HashSet

Если ты меняешь `Equals`/`GetHashCode` типа, после hotload ключи в словаре могут оказаться «не на своих местах» — движок предупредит. Лечится рестартом.

## Арраи и списки

Для массивов и `List<T>` есть быстрый путь, если тип — значимый и не содержит ссылочных полей:

```csharp
public record struct Pair( Vector3 Pos, int Id );

public int[] Ints;         // быстро
public Pair[] Pairs;        // быстро
public Vector3[] Vectors;   // быстро
public string[] Strings;    // медленно (ссылки)
public object[] Objects;    // медленно
```

## `[SkipHotload]`

Атрибут, который говорит hotload: **«не трогай это поле»**. Используй с **осторожностью** — если hotload пропустит поле, его значение останется от старой версии кода, что может вызвать странные баги.

Типичные легальные случаи:
- Кеш рефлексии.
- Константоподобные static поля.
- Static generic-поля.

## Практические советы

- ✅ Мелкие правки метода — спокойно сохраняй, hotload отработает мгновенно.
- ✅ После больших изменений (новые типы, иерархия) — **смотри консоль** на предмет ошибок.
- ✅ Если UI моргает, компоненты пропадают, странные `null` — **Restart Editor**.
- ✅ Если в проекте тест-сцена, которую ты каждый раз перезапускаешь, — не ленись, делай hotload.
- ❌ Не надейся, что hotload переподпишет тебе события — иногда надо делать это вручную.
