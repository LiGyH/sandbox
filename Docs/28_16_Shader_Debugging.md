# 28.16 — Отладка шейдеров

## Что мы делаем?

Закрываем Фазу 28 практическим блоком — **что делать, когда шейдер не работает**. Отладка GPU-кода устроена принципиально иначе, чем CPU: ты не можешь поставить breakpoint в `MainPs`. Зато у тебя есть инструменты графического дебага и набор стандартных приёмов.

## Симптомы и диагностика

### 1. Розовый/фиолетовый материал

**Что значит:** компилятор шейдера упал, движок подставил «error material».

**Что делать:**

- Открой консоль (`~`) и поищи сообщения вида `Failed to compile shader ...`.
- Открой `.shader` в редакторе кода — посмотри подсветку синтаксиса.
- Проверь, что `.shader_c` обновлён (рядом с `.shader`).
- Часто причина — опечатка `flot` вместо `float`, отсутствующий `;` или несовпадение semantic.

### 2. Полностью чёрный/белый/прозрачный

**Что значит:** шейдер скомпилирован, но логика возвращает не то.

**Что делать (бинарный поиск):** замени `MainPs` на возврат фиксированного цвета:
```hlsl
return float4( 1, 0, 0, 1 );  // должно быть красное
```
Если не красное — проблема в render states, blend, depth. Если красное — проблема дальше в твоей логике, режь её пополам и проверяй.

### 3. Параметр из C# «не доезжает»

**Что значит:** имя в `Attributes.Set("...")` не совпадает с `Attribute("...")` в шейдере, или тип не совпадает.

**Что делать:**

- Сверь имена побуквенно (регистр важен).
- Замени `Attribute(...)` на `Default(...)` со специфическим значением (например, `999`) и проверь, что оно «прошло» через шейдер.
- Если всё равно тишина — значит, `Attributes.Set` идёт **на другой материал/SceneObject**. Проверь, что ты пишешь именно туда.

### 4. Сетка/линии моргают на расстоянии

Не используешь `ddx/ddy` для anti-aliasing. См. [28.13 — snap_grid](28_13_SnapGrid_Shader.md).

### 5. Эффект «накапливается» каждый кадр

Где-то ты пишешь в backbuffer без сброса. Проверь `Stage`/`priority` — возможно, эффект применяется дважды или его источник = его же выходу.

## Полезные ConVars

Включаются в консоли:

| ConVar | Что показывает |
|---|---|
| `r_show_shader_compile_errors 1` | Громкие сообщения об ошибках компиляции (если выключены). |
| `r_print_shader_features` | Список features текущих шейдеров и их вариаций. |
| `r_print_active_render_passes` | Какие render-проходы и в каком порядке выполняются (полезно для пост-эффектов). |
| `mat_wireframe 1` | Wireframe-режим — видно сами меши. |
| `mat_normalmaps 0` | Отключить нормали — иногда баг именно в них. |
| `r_drawopaque 0` / `r_drawtranslucent 0` | Отключить непрозрачные / прозрачные — изолировать слой. |
| `r_show_postprocess 0` | Отключить пост-обработку целиком. |

> Точные имена ConVar’ов могут меняться между версиями s&box. Используй автодополнение по `r_show*` / `r_print*` / `mat_*`.

## RenderDoc / PIX — глубокая отладка

Когда «глазами не видно, что не так» — берёшь графический дебаггер.

### RenderDoc (рекомендуется для Windows/Linux)

1. Скачай с [renderdoc.org](https://renderdoc.org/).
2. **Launch Application** → выбери `sbox.exe` (или путь до игры).
3. Запусти Sandbox через RenderDoc.
4. В игре нажми **F12** (или горячую клавишу RenderDoc) — захватит один кадр.
5. Открой capture в RenderDoc.

Что увидишь:

- **Event Browser** — все draw call’ы и пост-проходы в порядке выполнения. Имена твоих `Blit(blit, "MyEffect")` будут как метки.
- Кликнув на проход — видишь **input** (RT/текстуры/uniforms), **output** (RT куда пишет), значения всех `Attribute(...)`.
- **Pixel History** — для конкретного пикселя на экране показывает, какие проходы его меняли и в каком порядке. Бесценно для «почему этот пиксель розовый».
- **Shader debugging** — можно прогнать VS/PS на одной вершине/пикселе и пошагово смотреть значения переменных.

### PIX (Microsoft, только Windows)

Альтернатива RenderDoc, более «корпоративная». Принципы такие же.

## Внутри-шейдерные отладочные приёмы

Ты не можешь поставить breakpoint, но можешь **возвращать значения как цвет**:

```hlsl
// проверить значение float-параметра
return float4( Strength.xxx, 1 );    // серый = значение

// проверить UV
return float4( i.vTexCoord, 0, 1 );  // должен быть градиент R→X, G→Y

// проверить, проходит ли ветка
return float4( 1, 0, 0, 1 );         // да — красным

// проверить float, мапнуть в виде heatmap
float v = saturate( something );
return float4( v, 1 - v, 0, 1 );     // от зелёного к красному
```

Этот «debug-вывод» — самый быстрый способ найти баг в формуле.

## Чек-лист «шейдер не работает»

1. Открыл консоль — нет ли ошибок компиляции?
2. Замени `MainPs` на `return float4(1,0,0,1);` — рисуется красным?
3. Имена `Attribute(...)` побуквенно сверены с `Attributes.Set("...")`?
4. `MODES` содержит правильный проход (`Forward()` для surface, `Default()` для post)?
5. `RenderState`-ы корректны (depth/blend)?
6. `.shader_c` обновлён рядом с `.shader`?
7. `Material.FromShader(...)` — путь точно правильный, не `shaders\postprocess\...` (бекслеши на Windows иногда едят)?
8. Эффект **включён** (компонент Enabled, GameObject Enabled, на правильной камере)?
9. Если ничего не помогло — RenderDoc/PIX.

## Подсказки на практике

- **Версионируй `.shader_c` в git.** Без него у коллег без локальной компиляции всё сломается.
- **Делай мини-тестовый шейдер** на каждый новый эффект — проще «hello world» отладить, чем сразу 200 строк.
- **Кэшируй RenderDoc-снимки** «как должно быть» в репозиторий (как png/screenshot) — потом легко сравнить.
- **При апгрейде s&box** перекомпилируй все свои `.shader` — иногда меняются стандартные includes.
- **Не оптимизируй до того, как заработало.** Сначала корректно, потом быстро.

## Результат

После этого этапа ты:

- ✅ Знаешь типичные симптомы и их причины.
- ✅ Умеешь использовать `r_show_*` / `r_print_*` ConVar’ы.
- ✅ Готов запустить RenderDoc на Sandbox и поймать капчу кадра.
- ✅ Владеешь приёмом «debug return float4(...)» внутри шейдера.
