# 27.06 — Устройство сетевого кода: tick, snapshot, RPC, [Sync]

## Что мы делаем?

Открываем «капот» сети s&box и смотрим, **что физически летает по проводам** между сервером и клиентом: с какой частотой, в каком формате, в каком порядке. Без этого знания нельзя ни масштабировать сервер на сотни игроков ([27.08](27_08_Scaling_Players.md)), ни выбрать паттерн под жанр ([27.07](27_07_Networking_Patterns.md)).

> 📚 База: [Facepunch/sbox-docs → networking](https://github.com/Facepunch/sbox-docs/tree/master/docs/networking) (toc.yml: networked-objects, ownership, rpc-messages, sync-properties, network-visibility, custom-snapshot-data, network-events).

## Большая картинка: 4 потока трафика

В s&box по сети идут **четыре** разных потока (см. [00.20](00_20_Networking_Basics.md)):

| Поток | Когда отправляется | Кто инициирует |
|---|---|---|
| **Snapshot** (`[Sync]` + transform) | каждый сетевой tick | сервер → клиенты |
| **RPC** | по событию (выстрел, чат…) | любой → любой (по правам) |
| **Spawn / Destroy** | при создании/удалении сетевого объекта | хост |
| **Ownership** | при `TakeOwnership`/`DropOwnership` | хост или владелец |

Тяжёлый на канал — **snapshot**. RPC обычно редкие. Spawn/ownership — единичные события.

## Tick rate vs UpdateRate

Из [00.08](00_08_Настройки_сети.md) знаем: в `Networking.config` есть `UpdateRate` (по умолчанию 30). Но физика идёт чаще (`Scene.FixedUpdate` ≈ 50 Hz). Что между ними?

```
Физика (FixedUpdate, 50 Hz):    | | | | | | | | | | | | | | | | |
Snapshot выдача (UpdateRate, 30): |   |   |   |   |   |   |   |
```

Сервер **считает физику чаще**, чем шлёт. Между snapshot-ами клиент **интерполирует** (плавно сдвигает прокси-объекты), чтобы не было «телепортов». См. [00.22](00_22_Ownership.md) про прокси.

> 💡 На быстрых играх (CS-подобный шутер) UpdateRate можно поднимать до 64 Hz. На MMO/RP — наоборот, опускать до 15–20 Hz, чтобы освободить канал. Об этом подробнее в [27.08](27_08_Scaling_Players.md).

## Snapshot: что в нём

В каждый tick сервер для **каждой пары (объект, клиент)** решает:

1. **Виден ли объект клиенту?** (см. `INetworkVisible` и `AlwaysTransmit` в [26.14](26_14_Network_Visibility.md)).
2. **Что у объекта изменилось** с прошлого snapshot-а?
   - Transform (position/rotation/scale) — если объект сетевой и шевелится.
   - Все `[Sync]`-свойства, у которых значение поменялось.
   - Custom snapshot data (см. ниже).
3. **Запаковать только дельту** и отправить.

Если ничего не изменилось — пакета по объекту **не будет вовсе**. Это и есть главная причина, почему «миллион пустых пропов» обычно не убивают сервер: они тихие.

### Custom Snapshot Data

Иногда [Sync]-полей не хватает: нужно отправить не «состояние», а серию событий (например, последние 4 выстрела за tick). Тогда в Facepunch-доках есть **Custom Snapshot Data** — ты можешь **дописать свои байты** в snapshot и распаковать на клиенте. Это путь, по которому делают «компактные» движения и события для больших режимов.

См. [Facepunch/sbox-docs → custom-snapshot-data.md](https://github.com/Facepunch/sbox-docs/blob/master/docs/networking/custom-snapshot-data.md).

## `[Sync]` — как это работает

`[Sync]` (см. [00.24](00_24_Sync_Properties.md)) — это автогенерация:

1. На **владельце** объекта (см. [00.22](00_22_Ownership.md)): любое присваивание помечает поле «грязным».
2. В следующий snapshot — поле летит остальным.
3. У получателей сеттер вызывается **через сетевой setter**, не локально.

Подробности:

| Атрибут | Что меняет |
|---|---|
| `[Sync]` | базовое — поле живёт у владельца, едет копиям |
| `[Sync( SyncFlags.FromHost )]` | поле меняет только хост (типичный вариант для серверной правды) |
| `[Sync( SyncFlags.Interpolate )]` | для плавных значений (скаляров) — клиент интерполирует |
| `[Sync( SyncFlags.Query )]` | поле активно «опрашивает» новое значение, а не «вылетает» |

> ⚠️ Каждое `[Sync]`-поле — **расход трафика**. На карликовом транспорте (10 игроков, 50 объектов) — пофиг. На MMO — каждое лишнее `[Sync] float Stamina` × 10000 объектов × 30 раз/с = десятки МБ/с. Отделяй «то, что должны видеть все» от «то, что меняется только локально».

## RPC — как устроен

Из [00.23](00_23_Rpc_Messages.md) знаем три атрибута:

| Атрибут | Куда летит |
|---|---|
| `[Rpc.Broadcast]` | всем подключённым |
| `[Rpc.Owner]` | только владельцу объекта |
| `[Rpc.Host]` | только хосту (типично — клиент → сервер с просьбой) |

Под капотом:

- RPC **сериализует аргументы** (поддерживаются примитивы, `Vector3`, `Rotation`, `string`, `GameObject`, ваши `[Serializable]`-типы).
- RPC **не гарантирует порядок** относительно snapshot-а: клиент может получить «выстрел» (`Rpc`) **до** того, как `Position` нового пуля-объекта приедет. Поэтому большие эффекты пускают **локально** клиентом, а не ждут пакет.
- RPC **может быть отложен**, если канал перегружен. Не ставь критичную игровую логику только на RPC — дублируй важное через `[Sync]`-поле.

### Канал: надёжный или нет?

Большинство RPC s&box идёт в **надёжном** канале (порядок и доставка гарантированы). Snapshot-ы — частично надёжные/нет (поле через 1–2 tick-а всё равно будет верным, последняя версия вытесняет предыдущие). Это значит:

- Для **частых** мелких событий (звуки выстрелов, эффекты крови) RPC — норм.
- Для **очень частых** (позиции пуль траектории) — лучше **не через RPC**, а либо через `[Sync]`-объект, либо через custom snapshot data.

## Сетевой объект: как «опубликовать»

Кратко (детали — [00.21](00_21_Networked_Objects.md)):

```csharp
var obj = SceneUtility.GetPrefabScene( prefab ).Clone();
obj.NetworkSpawn( connectionForOwner );
```

- Без `NetworkSpawn` объект существует **только локально** — его никто, кроме тебя, не видит.
- Параметр — кому отдать **владение**. Передашь `null` — владельцем будет хост.

## Owner и Proxy

- **Owner** — тот, чей клиент управляет объектом (читает ввод, правит `[Sync]`-поля).
- **Proxy** — все остальные. У них объект существует, но **не управляется** ими — только отрисовывается и интерполируется.

Главная проверка в коде: `if ( IsProxy ) return;` (см. [00.20](00_20_Networking_Basics.md)).

Передача владения:

```csharp
GameObject.Network.TakeOwnership();   // я хочу владеть
GameObject.Network.DropOwnership();   // я отдаю владение
GameObject.Network.AssignOwnership( connection ); // только хост
```

Зачем: для физических предметов, которыми **тащит** игрок (PhysGun, см. Фаза 8), ownership даётся на время — у клиента игрока будет «честная» физика без сетевых лагов.

## Network Visibility (PVS)

В [26.14](26_14_Network_Visibility.md) разобрано: `INetworkVisible` решает, входит ли объект в snapshot для конкретного клиента. Это **главный** механизм масштабирования. Без него на 256 игроков сервер задохнётся через 30 секунд.

## Сетевой бюджет: где упираемся

Условные ориентиры (порядок, не точные числа — зависят от железа и UpdateRate):

| Узкое место | Что съедает |
|---|---|
| **Полоса трафика** | (количество видимых объектов) × (размер snapshot-а) × UpdateRate × (число клиентов) |
| **CPU сервера** | физика + serialize snapshot-а + visibility-проверки |
| **CPU клиента** | deserialize snapshot-а + интерполяция + render |
| **Пинг (RTT)** | задержка между «нажал клавишу» и «увидел эффект» |

Ориентир по полосе: **20–80 КБ/с на клиента** для среднего шутера/sandbox. На RP с тысячью пропов — может быть 200+ КБ/с, и это уже больно. Снижают через `INetworkVisible`, реже UpdateRate, custom snapshot data.

## Network Events (для дебага)

В [00.25](00_25_Network_Events.md) — `INetworkListener`. На сервере это *главный* способ заметить «приехал/уехал», добавлять метрики:

```csharp
public class ConnectionMetrics : Component, Component.INetworkListener
{
    void INetworkListener.OnActive( Connection c )
    {
        Log.Info( $"[Net] +{c.DisplayName}, всего: {Connection.All.Count()}" );
        Metrics.Increment( "players.online" );
    }

    void INetworkListener.OnDisconnected( Connection c )
    {
        Log.Info( $"[Net] -{c.DisplayName}, играли {DateTime.UtcNow - c.ConnectedAt}" );
        Metrics.Decrement( "players.online" );
    }
}
```

## Что важно запомнить

- 4 потока трафика: snapshot, RPC, spawn/destroy, ownership. Самый тяжёлый — snapshot.
- Snapshot — **дельта** от прошлого, не полное состояние. Тихие объекты бесплатны.
- `[Sync]`-поле едет от **владельца** к остальным, в snapshot-овом канале.
- `[Rpc.Broadcast/Owner/Host]` — событийные сообщения, надёжные, но порядок относительно snapshot не гарантирован.
- `INetworkVisible` ([26.14](26_14_Network_Visibility.md)) — главный рычаг масштабирования.
- На больших режимах поднимай custom snapshot data, опускай UpdateRate, агрессивно скрывай объекты.
