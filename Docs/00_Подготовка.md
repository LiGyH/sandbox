## 1. Что такое s&box

**s&box** — это игровой движок нового поколения от студии **Facepunch** (создатели Garry's Mod и Rust). Движок построен на основе **Source 2** от Valve и предоставляет современную среду разработки игр.

Ключевые особенности:

- **Язык программирования — C#.** Вся игровая логика пишется на C#, одном из самых популярных и понятных языков программирования. Если вы никогда не программировали — не переживайте, мы будем разбирать каждую строку кода.
- **UI на Razor.** Пользовательский интерфейс (меню, индикаторы здоровья, чат, табло) создаётся с помощью **Razor** — технологии, похожей на HTML, но с возможностью вставлять C#-код прямо в разметку. Это аналог Microsoft Blazor.
- **Встроенный редактор.** s&box поставляется с визуальным редактором, в котором можно размещать объекты на карте, настраивать параметры и тестировать игру.
- **Мультиплеер из коробки.** Движок изначально рассчитан на сетевую игру — вам не нужно писать сетевой код с нуля, достаточно использовать встроенные атрибуты синхронизации.

## 3. Установка s&box

### Шаг 3.1. Получите доступ

s&box распространяется через **Steam**. На момент написания руководства движок находится в раннем доступе. Вам нужно:

1. Откройте **Steam** на своём компьютере.
2. Найдите **s&box** в магазине Steam и приобретите/добавьте его в библиотеку.
3. Нажмите **«Установить»** и дождитесь завершения загрузки.

### Шаг 3.2. Запустите редактор

После установки:

1. Запустите **s&box** из библиотеки Steam.
2. Откроется **лаунчер** — главное окно, из которого можно запускать игры, редактор и управлять проектами.
3. Перейдите в раздел **Editor** (Редактор) — это основное рабочее место разработчика.

### Шаг 3.3. Создайте новый проект

В редакторе:

1. Выберите **Create New Project** (Создать новый проект).
2. Укажите название — например, `sandbox`.
3. Выберите тип проекта — **Game**.
4. Редактор создаст структуру папок и файл проекта `.sbproj`.

> **Примечание:** Мы будем пересоздавать содержимое проекта с нуля, поэтому после создания проекта мы заменим все файлы нашими собственными. Но сам процесс создания проекта гарантирует, что s&box правильно его зарегистрирует.

### Шаг 3.4. Убедитесь, что всё работает

1. Откройте созданный проект в редакторе.
2. Нажмите **Play** — должна запуститься пустая сцена.
3. Если вы видите окно игры (пусть даже пустое) — всё установлено правильно.

## 4. Что мы будем создавать

Мы будем воссоздавать **Sandbox** — официальный игровой режим для s&box от Facepunch. Это **физическая песочница**, в которой игроки могут:

- **Спавнить пропы** (3D-модели) из огромной библиотеки объектов
- **Перемещать объекты** с помощью **PhysGun** (физическая пушка) — хватаете объект лучом и двигаете его в пространстве
- **Использовать ToolGun** (инструментальная пушка) с **30+ режимами** инструментов:
  - **Weld** — сваривание объектов
  - **Rope** — соединение верёвкой
  - **Thruster** — реактивный двигатель
  - **Wheel** — колесо (для создания машин)
  - **Hoverball** — парящий шар
  - **Balloon** — воздушный шар
  - **Elastic** — резинка
  - **Trail** — хвост/след
  - **Duplicator** — копирование построек
  - И многое другое
- **Стрелять** из различного оружия: пистолет (Glock, Colt 1911), автомат (M4A1), пистолет-пулемёт (MP5), дробовик, снайперская винтовка, монтировка, гранаты
- **Управлять NPC** — создавать учёных (Scientist), роллермайнов (Rollermine), боевых NPC
- **Строить конструкции** — машины, мосты, катапульты, дома — используя физические ограничения (constraints)
- **Общаться** через текстовый чат и голосовую связь
- **Сохранять и загружать** свои постройки
- **Настраивать пост-обработку** — виньетирование, блум, хроматическая аберрация, глубина резкости

Финальный результат будет **точной копией** оригинального Sandbox от Facepunch — те же файлы, та же логика, те же системы.

## 6. Как устроен движок s&box изнутри

Прежде чем писать код, важно понять ключевые концепции движка. Это фундамент, на котором строится вся игра.

### 6.1. Scene / GameObject / Component

Архитектура s&box похожа на **ECS** (Entity Component System), но с особенностями:

- **Scene** (Сцена) — корневой контейнер всего игрового мира. Всё, что существует в игре, находится в сцене.
- **GameObject** (Игровой объект) — базовая единица в сцене. Это «контейнер», который сам по себе ничего не делает, но может содержать компоненты. У каждого GameObject есть позиция, вращение и масштаб в мировом пространстве (Transform).
- **Component** (Компонент) — логика, прикреплённая к GameObject. Именно компоненты определяют поведение: `Player` — компонент, `BaseWeapon` — компонент, `Npc` — компонент.

Пример: игрок в Sandbox — это GameObject с компонентами:
- `Player` — логика здоровья, урона, смерти
- `PlayerController` — движение, прыжки, физика
- `PlayerInventory` — инвентарь оружия
- `PlayerFlashlight` — фонарик
- `PlayerData` — сетевые данные (ник, убийства, смерти)
- `SkinnedModelRenderer` — 3D-модель персонажа

Компоненты имеют жизненный цикл:
- `OnAwake()` — вызывается при создании
- `OnStart()` — вызывается после первого обновления
- `OnUpdate()` — вызывается каждый кадр
- `OnFixedUpdate()` — вызывается с фиксированной частотой (для физики)
- `OnDestroy()` — вызывается при уничтожении

### 6.2. GameObjectSystem

**GameObjectSystem** — это синглтон-система, которая живёт в сцене. В отличие от компонентов, которые прикреплены к конкретным GameObject'ам, системы существуют «глобально» на уровне сцены и не привязаны к какому-либо объекту.

Объявляется так:

```csharp
public sealed partial class GameManager : GameObjectSystem<GameManager>
{
    public GameManager( Scene scene ) : base( scene )
    {
        // Инициализация
    }
}
```

В Sandbox используются следующие GameObjectSystem'ы:

| Система | Назначение |
|---------|-----------|
| `GameManager` | Главный контроллер игры: подключение игроков, спавн, настройки |
| `Effects` | Визуальные эффекты: искры, кровь, взрывы |
| `CameraNoiseSystem` | Тряска и шум камеры |
| `CleanupSystem` | Очистка карты до исходного состояния |
| `SaveSystem` | Сохранение и загрузка построек |
| `UndoSystem` | Система отмены последнего действия |
| `ControlSystem` | Управление транспортом (стулья, машины) |
| `PostProcessManager` | Управление пост-обработкой |
| `BanSystem` | Банлист |
| `Hints` | Всплывающие подсказки |
| `FreeCamGameObjectSystem` | Свободная камера |

Обращение к системе из любого места кода:

```csharp
var manager = Scene.GetSystem<GameManager>();
// или через синглтон:
GameManager.Instance.SpawnPlayer( playerData );
```

### 6.3. Networking — сетевая модель

s&box использует модель **Host / Client** (Хост / Клиент):

- **Host** (Хост) — один из игроков, который запустил сервер. Он — «авторитет»: все важные решения (кто получил урон, где появится объект) принимает хост.
- **Client** (Клиент) — все остальные игроки, подключённые к хосту.

#### Синхронизация свойств: `[Sync]`

Атрибут `[Sync]` автоматически синхронизирует значение свойства между хостом и всеми клиентами:

```csharp
// Это свойство автоматически передаётся всем клиентам при изменении
[Sync] public int Kills { get; set; }

// SyncFlags.FromHost — только хост может менять значение
[Sync( SyncFlags.FromHost )] public float Health { get; set; } = 100;
```

В Sandbox `[Sync]` используется для: здоровья игрока, количества убийств и смертей, состояния оружия, положения NPC, состояния дверей и кнопок, и многого другого (25+ свойств).

#### Удалённые вызовы: `[Rpc.*]`

**RPC** (Remote Procedure Call) — удалённый вызов процедуры. Это способ вызвать метод на другой машине по сети:

- **`[Rpc.Host]`** — метод выполняется **только на хосте**. Клиент вызывает метод, но код выполняется на сервере. Используется для всех действий, требующих авторитета: спавн объектов, нанесение урона, изменение настроек.

```csharp
[Rpc.Host]
public void RequestSpawn( string prefabPath )
{
    // Этот код выполнится ТОЛЬКО на хосте,
    // даже если метод вызван клиентом
    var go = SceneUtility.Instantiate( prefabPath );
    go.NetworkSpawn();
}
```

- **`[Rpc.Broadcast]`** — метод выполняется **на всех** машинах (и хосте, и клиентах). Используется для визуальных эффектов, звуков, уведомлений в чате.

```csharp
[Rpc.Broadcast]
public void PlayExplosionEffect( Vector3 position )
{
    // Этот код выполнится на ВСЕХ подключённых клиентах
    Sound.Play( "explosion", position );
}
```

- **`[Rpc.Owner]`** — метод выполняется **только у владельца** объекта. Используется для персональных уведомлений, обновления UI конкретного игрока.

```csharp
[Rpc.Owner]
public void NotifyAmmoChanged( int newAmmo )
{
    // Выполнится только у владельца этого объекта
}
```

В Sandbox используется более **100 RPC-методов** — это основа всего сетевого взаимодействия.

### 6.4. Prefabs — шаблоны объектов

**Prefab** (Префаб) — это сохранённый шаблон GameObject'а со всеми его компонентами и настройками. Вместо того чтобы создавать сложный объект из кода каждый раз, вы сохраняете его как `.prefab`-файл и затем создаёте копии:

```csharp
// Создать копию префаба в мире
var go = SceneUtility.GetPrefabScene( prefabFile ).Clone();
go.WorldPosition = spawnPosition;
go.NetworkSpawn(); // Сделать видимым для всех по сети
```

Основные префабы Sandbox:

| Файл | Что содержит |
|------|-------------|
| `prefabs/engine/player.prefab` | Шаблон игрока со всеми компонентами |
| `prefabs/engine/spawnpoint.prefab` | Точка появления игроков |
| `prefabs/engine/spawn-progress.prefab` | Индикатор загрузки при спавне |
| `entities/sents/*.prefab` | Скриптовые сущности (стулья, лампы, NPC, динамит) |
| `entities/thruster/*.prefab` | Реактивные двигатели |
| `entities/wheel/*.prefab` | Колёса |
| `entities/balloon/*.prefab` | Воздушные шары |

### 6.5. GameResource — данные как ресурсы

**GameResource** — это способ хранить данные в виде файлов-ресурсов (аналог ScriptableObject в Unity). Вместо хардкода параметров в коде, вы создаёте ресурс, который можно редактировать в редакторе.

```csharp
[GameResource( "Scripted Entity", "sent", "Определение скриптовой сущности" )]
public class ScriptedEntity : GameResource, IDefinitionResource
{
    public string Title { get; set; }
    public Model Model { get; set; }
    public float Health { get; set; }
    // ...
}
```

Атрибут `[GameResource]` указывает движку:
1. Как называть ресурс в редакторе (`"Scripted Entity"`)
2. Какое расширение файла использовать (`"sent"`)
3. Описание для разработчика

В Sandbox используется **9 типов GameResource**:

| Класс | Расширение | Что описывает |
|-------|-----------|---------------|
| `ScriptedEntity` | `.sent` | Сущности (стулья, лампы, NPC) |
| `ScriptedEmitter` | `.spp` | Эмиттеры частиц |
| `ScriptedEmitterModel` | `.semit` | Модели эмиттеров |
| `ThrusterDefinition` | — | Параметры реактивных двигателей |
| `HoverballDefinition` | — | Параметры парящих шаров |
| `BalloonDefinition` | — | Параметры воздушных шаров |
| `WheelDefinition` | — | Параметры колёс |
| `LineDefinition` | — | Параметры линий/следов |
| `PostProcessResource` | — | Параметры пост-обработки |

### 6.6. Razor UI — пользовательский интерфейс

UI в s&box строится на **Razor** — технологии, сочетающей HTML-подобную разметку с C#-логикой. Если вы знакомы с веб-разработкой, это будет похоже на HTML + JavaScript, только вместо JavaScript — C#.

Файлы имеют расширение `.razor` и наследуют от `PanelComponent`:

```razor
@using Sandbox;
@using Sandbox.UI;
@inherits PanelComponent

<root>
    <div class="health-bar">
        <label>❤️ @Health</label>
        <div class="bar" style="width: @(Health)%"></div>
    </div>
</root>

@code {
    public float Health { get; set; } = 100f;

    protected override void OnUpdate()
    {
        var player = Scene.GetAllComponents<Player>()
            .FirstOrDefault( x => x.Network.IsOwner );

        if ( player is not null )
            Health = player.Health;
    }
}
```

Стилизация выполняется через `.scss`-файлы (аналог CSS).

В Sandbox **65+ Razor-файлов**, реализующих:

- **Чат** (`Chat.razor`) — текстовые сообщения между игроками
- **Табло** (`Scoreboard.razor`) — список игроков с убийствами и смертями
- **Инвентарь** (`Inventory.razor`) — панель оружия
- **Спавн-меню** (`SpawnMenu/`) — огромное меню для создания пропов, сущностей, дупликатов
- **Информация об инструменте** (`ToolInfo/`) — описание текущего инструмента ToolGun
- **Индикаторы здоровья** (`Vitals/`) — здоровье и броня
- **Фид убийств** (`Feed.razor`) — лента уничтожений
- **Контекстное меню** (`ContextMenu/`) — меню правой кнопки мыши
- **Одевалка** (`Dresser/`) — настройка внешнего вида персонажа
- **Фейс-позер** (`FacePoser/`) — управление мимикой NPC

## 8. Важные правила

Пожалуйста, запомните несколько правил, которые действуют на протяжении всего руководства:

### 8.1. После каждого шага можно компилировать и тестировать

Каждый шаг спроектирован так, чтобы после его завершения проект **компилировался без ошибок** и мог быть запущен. Вы сможете увидеть результат своей работы после каждого этапа — это важно для мотивации и для отладки.

### 8.2. Каждый шаг строится на предыдущем

Шаги выстроены в строгом порядке зависимостей. Нельзя перейти к шагу 09 (PhysGun), не завершив шаги 01–08, потому что PhysGun зависит от системы оружия, инвентаря, игрока и GameManager.

### 8.3. UI создаётся вместе с системами

Мы **не** откладываем весь UI на конец. Базовые элементы интерфейса (чат, табло, инвентарь, информация об инструменте) создаются в шагах 16–17, но некоторые UI-элементы, тесно связанные с конкретной системой, создаются вместе с ней. Например, UI прицела создаётся вместе с оружием.

### 8.4. Весь код — точная копия оригинала

Мы не «упрощаем» и не «адаптируем» код. Каждый файл, каждый класс, каждый метод — это **точная копия** из оригинального репозитория Sandbox от Facepunch. Это гарантирует:

- Код работает — он уже проверен в продакшене
- Вы изучаете реальные паттерны разработки
- Результат идентичен оригиналу

### 8.5. Не бойтесь кода, который пока непонятен

В некоторых файлах вы встретите конструкции, которые будут разобраны в более поздних шагах. Например, в `Player.cs` может быть ссылка на `PlayerInventory`, который мы создадим позже. Это нормально — просто следуйте инструкциям, а понимание придёт по мере прохождения руководства.
