# 28.10 — Шейдеры: общая картина

## Что мы делаем?

Открываем подфазу **28.B — Шейдеры**. Шейдеры — это **программы, которые считают каждый пиксель** на экране (а заодно и каждую вершину меша). В s&box у тебя есть полный доступ к написанию своих шейдеров на HLSL — от простых materials до пост-эффектов на весь экран.

> 🧭 Под этой темой много страха («шейдеры — это сложно»). На деле в s&box стандартный материал (`vr_complex.shader` или `vr_standard.shader`) покрывает 90% случаев. Свои шейдеры пишут только когда нужен **уникальный эффект** — снап-сетка, sniper scope, вода, огонь, голограмма.

## Конвейер: от .shader до пикселя

```
┌────────────────┐    компиляция    ┌──────────────────┐
│  *.shader      │  ───────────────▶│  *.shader_c      │
│  (HLSL+метa,   │                  │  (бинарь, готов  │
│   ты пишешь)   │                  │   к загрузке)    │
└────────────────┘                  └──────────────────┘
                                             │
                                             ▼
                              ┌──────────────────────────────┐
                              │     Material  (*.vmat)       │
                              │  выбирает шейдер,            │
                              │  задаёт параметры/текстуры   │
                              └──────────────────────────────┘
                                             │
                                             ▼
                       ┌──────────────────────────────────────┐
                       │ Render-проход GPU                    │
                       │  Vertex Shader (VS)  → меш в clip-sp │
                       │  Pixel  Shader (PS)  → цвет пикселя  │
                       └──────────────────────────────────────┘
```

- `.shader` — **исходник**. Текст. HLSL + метаданные блоками (`HEADER`, `MODES`, `COMMON`, `FEATURES`, `VS`, `PS`).
- `.shader_c` — **скомпилированный** артефакт. Появляется автоматически рядом с `.shader` после билда. В git его обычно тоже коммитят, чтобы у других разработчиков всё работало без локальной компиляции.
- `.vmat` — **материал**: ссылается на конкретный `.shader` и задаёт значения его параметров (цвет, текстуры, числа).

## Где живут шейдеры в Sandbox

```
Assets/shaders/
├── postprocess/
│   ├── sniper_scope.shader     ← пост-эффект для прицела снайперки
│   └── sniper_scope.shader_c
├── snap_grid.shader            ← оверлей сетки для Stacker/SnapGrid
└── snap_grid.shader_c
```

Это **весь** список «своих» шейдеров проекта. Всё остальное использует стандартные шейдеры движка (`vr_complex`, `simple_2d` для UI, `vr_unlit` и т.д.).

В коде они подгружаются так:

```csharp
// Code/Weapons/Sniper/SniperScopeEffect.cs:18
_cachedMaterial ??= Material.FromShader( "shaders/postprocess/sniper_scope.shader" );
```

`Material.FromShader(path)` — создать «безымянный» материал на основе шейдера. Дальше параметры шейдера ставишь через `Attributes.Set(...)`.

## Категории шейдеров

| Категория | Для чего | Пример в Sandbox |
|---|---|---|
| **Surface** (поверхностные) | Применяются на меше через `.vmat`. Считают цвет поверхности с учётом света. | стандартные `vr_complex` (используется во всех `.vmat` ассетов) |
| **Forward / Custom** | Применяются на меше, но без полной PBR-модели — рисуем что-то «своё». | `snap_grid.shader` (оверлей-сетка без освещения) |
| **PostProcess** | Применяются **на весь экран** после рендера сцены. На входе — backbuffer (готовая картинка), на выходе — модифицированная картинка. | `sniper_scope.shader` (расфокус, виньетка прицела) |
| **Compute** | Чисто вычислительный шейдер на GPU без рендера. | в Sandbox не используется |
| **UI** | Для отдельных Razor-панелей через `style.background`. | не используется напрямую |

## Forward vs PostProcess

Это две главных модели применения, важно различать:

### Forward (поверхностный)

- Меш отрисовывается с твоим шейдером.
- Каждый пиксель **меша** проходит через твой PS.
- Параметры приходят из `.vmat` или `SceneObject.Attributes.Set(...)`.
- Пример: `snap_grid.shader` рисует прозрачный квадрат на месте грани объекта со светящейся сеткой.

### PostProcess

- Сцена уже отрендерена в backbuffer.
- Делается **полно-экранный** проход твоего шейдера: берём текстуру backbuffer, считаем каждый пиксель экрана, пишем результат.
- Подключается через `BasePostProcess` + `BlitMode` (см. 28.14 / 28.15).
- Пример: `sniper_scope.shader` берёт картинку → размывает по краям → возвращает обратно.

## Минимальный «hello world» шейдер

```hlsl
HEADER
{
    Description = "Hello shader";
}

MODES
{
    Forward();          // рисуется в форвард-проход
}

COMMON
{
    #include "common/shared.hlsl"
}

struct VertexInput
{
    float3 vPositionWs : POSITION < Semantic( PosXyz ); >;
};

struct PixelInput
{
    float4 vPositionPs : SV_Position;
};

VS
{
    PixelInput MainVs( VertexInput i )
    {
        PixelInput o;
        o.vPositionPs = Position3WsToPs( i.vPositionWs );
        return o;
    }
}

PS
{
    float4 MainPs( PixelInput i ) : SV_Target0
    {
        return float4( 1, 0, 1, 1 ); // ярко-розовый — «я работаю»
    }
}
```

Этого достаточно для шейдера, рисующего меш сплошным цветом. Подробный разбор синтаксиса — в [28.11](28_11_Shader_Anatomy.md).

## Как ассет понимает, какой шейдер использовать

В `.vmat`-файле есть строка:

```vmat
"Shader" "shaders/snap_grid.shader"
```

Параметры с тем же именем, что объявлены в `PS` блоке шейдера через `Attribute("...")`, автоматически становятся ползунками/полями ввода в Material Editor.

## Подсказки на практике

- **Не пиши свой шейдер для обычного PBR-материала.** Бери `vr_complex` — он даёт normal/metalness/roughness/AO/emission и т.д.
- **Свой шейдер — для эффекта, которого нет в стандартных.** Снап-сетка, scope, голограмма, сканлайны.
- **`.shader_c` коммить вместе с `.shader`** — иначе у коллег без компилятора всё сломается.
- **Имена шейдеров регистро-чувствительны** в путях `Material.FromShader` — точно как файл на диске.
- **Шейдеры компилируются на стороне разработчика**, не у игрока. Игрок получает только `.shader_c`.
