# 26.20 — ActionGraph: введение

## Что мы делаем?

`ActionGraph` — встроенная в s&box **система визуального скриптинга** (как Blueprints в UE или Bolt в Unity). Соединяешь блоки (узлы) проводами, описываешь логику без кода. Полезно для дизайнеров, прототипов, скриптов уровня и точечных «приклеиваний» поверх кода.

## Зачем нужен ActionGraph

- Дизайнер уровня хочет сделать **триггерную дверь** «нажал кнопку → дверь открылась через 2 сек → сыграл звук» — без программиста.
- Программист хочет дать дизайнеру **настраиваемое поведение** оружия (рикошеты, разные звуки), не пересобирая код.
- Прототипировать **редкое одноразовое событие** на уровне (катсцена, скриптовая ловушка) — быстрее графом, чем компонентом.
- Расширять компоненты **точечной C#-логикой** через event-делегаты (`Action`/`Func` в `[Property]`).

## Как устроен граф

| Понятие | Что это |
|---|---|
| **Root node** | стартовая точка, посылает сигнал, когда граф запускается |
| **Action node** (синий) | выполняет действие в порядке (поставить значение, играть звук, дождаться) |
| **Expression node** (зелёный) | вычисляет значение (`a + b`, `x > 0`, нормаль вектора) |
| **Link** | провод между узлами; синие — сигналы (control flow), зелёные — данные |
| **Variable** | хранилище значения внутри графа |

Сигнал течёт от `Root` дальше по action-узлам в указанной последовательности; expression-узлы тянут данные «вверх» к узлам, которым нужны входные параметры.

## Где встречается ActionGraph

1. **На компонентах с событиями.** Например, у триггерного коллайдера в инспекторе видишь поле `OnTriggerEnter` — это слот для графа. Добавил граф — описал реакцию.
2. **Action Resource** — `*.action`-файл, отдельный ассет с графом. Можно ссылаться из нескольких мест.
3. **`Actions Invoker`-компонент** — обёртка, запускающая граф, удобно для хук-точек уровня.
4. **C# `[Property] Action MyEvent { get; set; }`** — поле компонента типа делегата автоматически становится «слотом» для графа в инспекторе.

## Создать поле под граф в C#

```csharp
public sealed class Door : Component
{
    [Property] public Action OnOpened { get; set; }
    [Property] public Action<float> OnDamaged { get; set; }

    public void Open()
    {
        // ...
        OnOpened?.Invoke();
    }
}
```

В инспекторе у `Door` появятся слоты `OnOpened` и `OnDamaged` — дизайнер сможет нарисовать туда граф (играть звук, выпустить частицы, открыть свет).

## Минимальный граф «открыть дверь — сыграть звук»

```
[Root: OnOpened] ─▶ [Sound.Play( "door_open.sound", self.WorldPosition )]
                  └▶ [Set Light.Enabled = true]
```

Без C#. Дизайнер тащит узлы из палитры, соединяет, сохраняет.

## Граф ↔ C# (взаимный вызов)

- **Из кода в граф:** просто вызови делегат (`OnOpened?.Invoke()`).
- **Из графа в код:** добавь в C# атрибут `[ActionGraphNode]` к статическому методу или объяви `[ActionGraphIncluded]` на компоненте — метод появится в палитре графов.

```csharp
[ActionGraphNode( "myproject.dohealsplash" )]
public static void DoHealSplash( Vector3 pos, float strength )
{
    Effects.HealSplash( pos, strength );
}
```

Этот метод теперь доступен любому графу в проекте.

## Variables (переменные)

Графы могут хранить состояние. Объявляй переменные в редакторе графа: имя, тип. Используй узлы `Get/Set <var>`. Хранятся **в экземпляре графа**, не глобально.

## Когда граф предпочтительнее кода

| Ситуация | Лучше |
|---|---|
| Один-два простых шага (играть звук, поставить значение) | граф |
| Цикл по списку 1000 элементов | C# |
| Логика, требующая отладчика и брейкпоинтов | C# |
| Точечная «обвеска» дизайнером | граф |
| Полностью алгоритмический NPC-AI | C# |
| UI-эффект на нажатие кнопки | граф |

Не пытайся писать **весь геймплей** в графе — он не для этого. Граф хорош для **точечного клея**.

## Производительность

ActionGraph выполняется через интерпретатор и медленнее чистого C#. Не страшно для редких событий (`OnOpened`, `OnDestroy`), но не клади его в `OnUpdate` тысячи раз.

## Что важно запомнить

- ActionGraph = визуальный скриптинг, не замена коду, а его дополнение.
- Граф запускается из root-узла; узлы бывают action (поток) и expression (данные).
- Поля C# типа `Action`/`Func` автоматически становятся слотами для графов.
- Свои методы можно открыть для графа через `[ActionGraphNode]` / `[ActionGraphIncluded]`.
- Переменные графа хранят его локальное состояние.
- Граф = клей для дизайнера; код = тяжёлая логика для программиста.
